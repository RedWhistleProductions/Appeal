Appeal Runtime

Appeal is designed as a general-purpose programming language built around command streams. A command stream can come from a file today and can later come from a network data source, generated script, package system, editor, replay, or other provider.

Runtime Paths

Path Purpose
./Resources Default resource root.
./Resources/Appeal Default folder for Appeal scripts.
./Resources/Modules Default folder for built dynamic modules and their plugins.

Startup Flow

The current executable starts by running Main.Appeal. That script normally runs Config.Appeal, which loads the default modules and plugins.

Run Config
Done

Later, the app can accept an alternate startup script for testing or launching directly into a game or tool.

Core Commands

Command Arguments Behavior
Run script Loads and switches to another script from Resources/Appeal.
Return none Returns from the current data source to the previous one.
Done none Stops the current run loop.
Print text Writes text to the terminal.
Add_Dynamic_Module module plugin Loads a module binary, registers its interpreter, and initializes the named plugin.
Delay seconds Pauses execution for an integer number of seconds.
Set_Var name value Stores a variable in the data manager.
Del_Var name Removes a variable from the data manager.

Modules And Plugins

Appeal modules are platform-independent command dictionaries. Plugins are platform-specific implementations loaded by those modules. Appeal scripts call module commands; the module parses arguments and forwards work to the plugin.

Add_Dynamic_Module Audio Test_Audio
Audio Load_Play_List PlayAll.txt
Audio Play_Song "Folk Song"

Design Direction

The long-term goal is to keep the Appeal language universal while letting each platform use optimized native plugins. Bananas will be a game-focused Appeal distribution, and a future browser can use the same module/plugin architecture with stricter security and responsive GUI behavior.
